## 2. 总体流程

```text
Notion / docs/项目总控.md
  确认阶段目标、当前进度、重要决策和下一步
        ↓
docs/
  固化产品、业务、页面、权限、数据库、API 资料
        ↓
GitHub Issues / Projects
  拆任务卡并管理状态
        ↓
Codex
  读取 AGENTS.md 和 docs/，执行单个任务
        ↓
Git 分支 + Pull Request
  保存本次修改，关联任务卡
        ↓
GitHub Actions
  自动检查代码质量
        ↓
腾讯云测试环境 / 预览环境
  生成页面预览或测试访问链接
        ↓
Robert 验收
  确认功能、页面和业务规则
        ↓
合并 PR，更新任务和文档
```

`AGENTS.md` 用于规定 Codex 的工作方式：

- 测试和验收要求

固定执行要求写入 `AGENTS.md` 后，Robert 不需要每次重复说明。

## 4. 任务看板

本地 `docs/任务清单.md` 和 GitHub Project #1 使用同一套状态：

| 状态 | 含义 |
|---|---|
| Backlog | 后续再做，尚未进入可执行队列 |
| Ready | 前置条件已满足，可以开始 |
| In Progress | 正在执行 |
| Review | 等待 Robert 验收或 PR 审核 |
| Blocked | 前置任务或必要条件尚未满足 |
| Done | 已完成 |

2026-07-19 已通过 GitHub CLI 将 Project #1 的原 `Todo / In Progress / Done` 扩展为以上六种状态，避免本地与看板使用两套口径。

任务推进规则：

1. 只有资料清楚、验收标准明确的任务在本地任务清单中标为 `Ready`。
2. Codex 每次只执行一个 `Ready` 任务。
3. GitHub Project 中对应卡片开始执行时从 `Ready` 移到 `In Progress`。
4. 功能完成并提交给 Robert 时进入 `Review`；验收通过并合并 PR 后进入 `Done`。
5. 尚未进入近期计划的任务使用 `Backlog`；前置条件不满足时使用 `Blocked`。

## 6. 单个任务执行流程

每个任务按以下流程执行：

1. Robert 从 `docs/项目总控.md` 或 `docs/任务清单.md` 选择一个 `Ready` 任务，并在 GitHub Project 中找到对应 Issue。
2. Robert 把 GitHub Issue 编号发给 Codex。
3. Codex 读取 `AGENTS.md` 和相关 `docs/` 文件。
4. Codex 确认任务范围和不做范围。
5. Codex 创建或使用对应任务分支。
6. Codex 修改代码或文档。
7. Codex 运行必要检查。
8. Codex 创建或更新 Pull Request。
9. GitHub Actions 自动检查。
10. 腾讯云测试环境生成预览或测试访问链接；Vercel 仅作为临时备选。
11. Codex 汇报结果。
12. Robert 根据预览链接或本地运行结果验收。
13. 验收通过后合并 PR。
14. 任务进入 `Done`。
15. 如有重要规则变化，同步更新 `docs/` 和 Notion。

## 7. Robert 给 Codex 的指令

执行已有任务时：

```text
请执行 GitHub Issue #编号。

执行要求：
- 先读取 AGENTS.md 和相关 docs/。
- 只做这个 Issue 的范围。
- 如果需求不清楚，先列出需要我确认的问题。
- 完成后运行必要检查，并创建或更新 PR。
- 最后按第 8 节格式汇报。
```

创建新任务卡时：

```text
请把“功能名称”拆成 GitHub Issue 草稿。

要求：
- 基于当前 docs/ 输出。
- 按第 5 节 Issue 格式输出。
- 如果信息不够，先列出需要我确认的问题。
```

要求阶段总结时：

```text
请根据当前 GitHub Issues、PR 和 docs/，总结当前阶段进度。

请包含：
- 已完成
- 正在做
- 待确认
- 风险/阻塞
- 建议下一步
```

临时补充要求时：

```text
补充要求：
- ...
```
