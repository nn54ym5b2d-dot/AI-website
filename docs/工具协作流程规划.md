# 工具协作流程规划

版本：v1.3  
日期：2026-06-17  
适用项目：源素库 / 虚拟素材网站 MVP，以及后续 App、微信小程序扩展  
使用对象：Robert 和 Codex  

## 1. 使用工具

| 工具 | 用途 |
|---|---|
| Notion | 项目总控、阶段目标、重要决策、进度总结 |
| GitHub 仓库 | 存放代码、文档、修改历史和 Pull Request |
| `docs/` | 存放开发资料，作为 Codex 执行任务的主要依据 |
| `AGENTS.md` | 存放 Codex 工作规则 |
| GitHub Issues | 存放具体任务卡 |
| GitHub Projects | 查看任务状态和推进进度 |
| GitHub Actions | 自动运行测试、构建、代码检查 |
| Codex | 执行开发、修改、测试、文档维护 |
| 腾讯云 | 中国大陆生产部署、备案、腾讯云 COS 文件存储、测试环境、上线前验收 |
| Vercel | 仅作为早期海外预览或临时演示备选，不作为中国大陆生产部署默认方案 |
| Apifox | 后期 API 文档、调试、Mock、接口测试 |

## 2. 总体流程

```text
Notion
  确认阶段目标和重要决策
        ↓
docs/
  固化产品、业务、页面、权限、数据库、API 资料
        ↓
GitHub Issues / Projects
  拆任务卡并管理状态
        ↓
Codex
  读取 AGENTS.md 和 docs/，执行单个任务
        ↓
Git 分支 + Pull Request
  保存本次修改，关联任务卡
        ↓
GitHub Actions
  自动检查代码质量
        ↓
腾讯云测试环境 / 预览环境
  生成页面预览或测试访问链接
        ↓
Robert 验收
  确认功能、页面和业务规则
        ↓
合并 PR，更新任务和文档
```

## 3. 资料位置

### 3.1 Notion

Notion 用于项目总控：

- 项目简介
- 当前版本目标
- 阶段计划
- 重要决策
- 当前进度
- 待确认问题
- 阶段总结

### 3.2 GitHub 仓库

GitHub 仓库用于工程管理：

- 网站代码
- 后台代码
- 数据库相关文件
- 测试代码
- 配置文件
- Pull Request 记录
- GitHub Actions 检查记录
- `docs/`
- `AGENTS.md`

### 3.3 `docs/`

`docs/` 用于存放 Codex 开发时必须读取的项目资料：

| 文件 | 用途 |
|---|---|
| `docs/PRD.md` | 产品需求和版本范围 |
| `docs/业务规则.md` | 上传、认证、订单、退款、授权规则 |
| `docs/页面清单.md` | 前台、上传者中心、后台页面列表 |
| `docs/角色权限.md` | 用户、上传者、管理员、外部观察员权限 |
| `docs/技术方案.md` | 技术栈、系统模块、部署方式 |
| `docs/数据库设计.md` | 数据表、字段和关系 |
| `docs/API.md` | 后端接口说明 |
| `docs/测试验收清单.md` | 核心流程和验收场景 |
| `docs/决策记录.md` | 重要决策及原因 |
| `docs/工具协作流程规划.md` | 当前工具协作流程 |

### 3.4 `AGENTS.md`

`AGENTS.md` 用于规定 Codex 的工作方式：

- 执行任务前读取哪些文件
- 如何控制任务范围
- 如何处理不清楚的需求
- 哪些操作需要先询问 Robert
- 完成后如何汇报
- 测试和验收要求

固定执行要求写入 `AGENTS.md` 后，Robert 不需要每次重复说明。

## 4. 任务看板

GitHub Projects 使用以下状态：

| 状态 | 含义 |
|---|---|
| Backlog | 后续再做 |
| Ready | 已准备好，可以开始 |
| In Progress | 正在执行 |
| Review | 等待检查或 Robert 验收 |
| Done | 已完成 |

任务推进规则：

1. 只有资料清楚、验收标准明确的任务进入 `Ready`。
2. Codex 每次只执行一个 `Ready` 任务。
3. 代码任务完成后进入 `Review`。
4. Robert 验收通过并合并 PR 后进入 `Done`。

## 5. GitHub Issue 格式

每张任务卡使用以下结构：

任务卡一般由 Codex 根据 `docs/` 生成，Robert 负责确认、补充或修改。

```text
任务目标：

相关资料：

关联需求编号：

本次范围：

不做范围：

影响范围：

验收标准：

测试要求：

备注：
```

## 6. 单个任务执行流程

每个任务按以下流程执行：

1. Robert 从 GitHub Project 选择一个 `Ready` 任务。
2. Robert 把 GitHub Issue 编号发给 Codex。
3. Codex 读取 `AGENTS.md` 和相关 `docs/` 文件。
4. Codex 确认任务范围和不做范围。
5. Codex 创建或使用对应任务分支。
6. Codex 修改代码或文档。
7. Codex 运行必要检查。
8. Codex 创建或更新 Pull Request。
9. GitHub Actions 自动检查。
10. 腾讯云测试环境生成预览或测试访问链接；Vercel 仅作为临时备选。
11. Codex 汇报结果。
12. Robert 根据预览链接或本地运行结果验收。
13. 验收通过后合并 PR。
14. 任务进入 `Done`。
15. 如有重要规则变化，同步更新 `docs/` 和 Notion。

## 7. Robert 给 Codex 的指令

执行已有任务时：

```text
请执行 GitHub Issue #编号。

执行要求：
- 先读取 AGENTS.md 和相关 docs/。
- 只做这个 Issue 的范围。
- 如果需求不清楚，先列出需要我确认的问题。
- 完成后运行必要检查，并创建或更新 PR。
- 最后按第 8 节格式汇报。
```

创建新任务卡时：

```text
请把“功能名称”拆成 GitHub Issue 草稿。

要求：
- 基于当前 docs/ 输出。
- 按第 5 节 Issue 格式输出。
- 如果信息不够，先列出需要我确认的问题。
```

要求阶段总结时：

```text
请根据当前 GitHub Issues、PR 和 docs/，总结当前阶段进度。

请包含：
- 已完成
- 正在做
- 待确认
- 风险/阻塞
- 建议下一步
```

临时补充要求时：

```text
补充要求：
- ...
```

## 8. Codex 完成后的汇报格式

```text
已完成：

修改文件：

PR / 预览链接：

如何验收：

测试结果：

风险/注意：

建议下一步：
```

## 9. 代码质量检查

GitHub Actions 默认运行：

```text
npm run lint
npm run typecheck
npm run test
npm run build
```

实际命令根据项目技术栈调整。

任务完成标准：

- 代码能运行。
- 构建能通过。
- 自动检查通过。
- 关键功能可验收。
- 文档需要更新时已更新。
