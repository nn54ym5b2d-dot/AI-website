# 测试验收清单

版本：v1.8
日期：2026-07-15

## 1. 使用方式

每完成一个开发任务，至少检查对应功能是否满足本文件的验收点。若暂时没有代码，本文件作为后续开发的验收依据。

## 2. 产品资料验收

- `docs/PRD.md` 能准确描述产品定位、目标用户和 MVP 范围。
- `docs/业务规则.md` 覆盖上传、认证、审核、购买、授权、收益和状态规则。
- `docs/页面清单.md` 能指导前台、个人中心和后台页面搭建。
- `docs/角色权限.md` 能明确不同角色能做什么、不能做什么。
- `docs/决策记录.md` 区分已确认和待确认问题。

## 3. 上传者流程验收

- 上传者不能无邀请码激活上传权限。
- 上传者可以创建人物、物件/道具、场景素材。
- 上传者可以上传原文件和预览图。
- 人物素材需要上传必要证明材料；物件/道具和场景素材不强制要求必要证明材料。
- 提交素材时进入认证上传费支付流程。
- 初审未通过时可以记录退款。
- 初审通过后可以进入认证状态。
- 认证完成后，管理员可以录入证书编号、凭证并保存认证证书文件。
- 所有素材，包括人物/物件/场景，必须获得认证证书后才可上架。

## 4. 购买下载流程验收

- 用户可以浏览、搜索、筛选素材。
- 用户可以查看素材详情、价格、认证状态和预览信息。
- 首版人物和场景素材统一显示 50 元，物件/道具素材统一显示 10 元。
- 未登录用户不能直接购买。
- 一个订单可以包含多个素材。
- 支付成功后，每个素材生成独立授权记录。
- 支付成功后，用户可以获取默认有效 365 天的下载入口；多角度原文件以一个 ZIP 素材包交付。
- ZIP 文件清单覆盖该素材全部有效原文件，且不能混入其他素材文件。
- 原始文件地址不能直接公开。
- 下载行为有记录。

## 5. 授权流程验收

- 授权记录包含购买用户、素材编号、订单编号、授权说明、授权时间和认证状态。
- 用户可以在个人中心查看自己的授权记录。
- 后台可以查看授权记录。
- 授权永久有效，不自然到期；授权状态支持有效、已撤销。

## 6. 收益流程验收

- 支付成功后，上传者收益记录增加。
- 首版每笔素材购买按上传者 80%、平台 20%、外部观察员 0% 记录收益。
- 上传者只能查看自己的收益。
- 管理员可以查看全部收益记录。
- 财务管理员可以查看收益和支付记录。
- 财务管理员可以发起和处理退款，且操作进入日志。
- 素材购买退款只能选择完整订单明细；拆分单个明细金额时必须拒绝，退款成功只撤销被退款明细的授权和收益。
- 第一版不要求自动提现。

## 7. 后台权限验收

- 超级管理员拥有全部后台权限；购买和上传仍要求对应基础角色。
- 运营管理员不能修改角色和系统级设置。
- 财务管理员不能审核素材、修改素材内容、录入认证信息或修改价格。
- 外部观察员只能只读查看平台上传量、下载量、收益汇总和合作方分成相关数据。
- 外部观察员首版分成比例和金额显示为 0。
- 外部观察员不能导出任何后台数据。
- 外部观察员不能查看用户手机号、邮箱、证件信息、支付账户、完整支付流水号、素材原文件地址或下载原始链接。
- 外部观察员可按人物、物件/道具、场景查看上传、上架、认证和下载汇总，但不能下钻到素材明细。
- 关键后台操作进入操作日志。

## 8. 安全与合规验收

- 原始素材文件不公开暴露。
- 用户下载入口默认有效 365 天；每次实际下载使用短时 COS 签名 URL。
- 手机号/邮箱注册和微信首次注册都必须保存所接受的有效条款版本；未接受时不能创建新用户。
- 实名认证不属于 MVP；系统不得收集或伪装保存未定义的实名资料。
- 后台接口必须检查角色权限。
- 敏感信息不向外部观察员展示。
- 微信支付和支付宝回调需要验签。
- 不在代码中写入密钥、token、密码。

## 9. Robert 手动验收建议

每个阶段完成后，Robert 可以按以下方式验收：

1. 先看页面是否符合业务理解。
2. 再按上传者、购买用户、管理员、外部观察员分别走一遍流程。
3. 检查不该看到的信息是否被隐藏。
4. 检查不该操作的按钮是否不可用或不可见。
5. 检查订单、授权、收益记录是否正确产生。

## 10. T008 API 接口合同验收

- 核心业务接口统一使用 `/api/v1` 前缀，成功和错误响应包含 `requestId`。
- 金额使用整数分，时间使用 ISO 8601 UTC，列表接口使用受限游标分页。
- Web 登录令牌只保存在 `HttpOnly`、`Secure`、`SameSite=Lax` Cookie，不写入 `localStorage`。
- Cookie 鉴权的修改请求同时校验来源和 CSRF token；未登录返回 `401`，角色或资源归属不足返回 `403`。
- 创建订单、发起支付、提交审核和退款接口要求 `Idempotency-Key`，重复请求不能生成重复业务记录。
- 支付/退款回调使用第三方事件标识去重，不依赖客户端 `Idempotency-Key`。
- 手机号/邮箱验证码、会话、上传意图、幂等和回调事件只保存必要哈希或摘要，不保存明文敏感值。
- 上传者先从应用服务申请短期签名地址，再直传私有 COS，最后调用完成接口由服务端核验对象。
- Next.js 应用服务器不接收素材文件正文；API 不单独返回 COS bucket、region 或 object key。
- 支付和退款回调基于原始请求体验签，并校验商户号、订单号、金额、币种和重复事件。
- 素材购买支付成功后，订单、授权、365 天平台下载入口和收益记录在事务中幂等生成。
- 365 天指平台下载资格；实际 COS 签名 URL 每次下载时短时生成，建议不超过 10 分钟。
- 下载前重新校验用户、授权、入口、文件归属和有效期，且历史接口不返回 COS URL 或下载 token。
- 后台审核、上架、认证、退款、权限、设置和授权撤销均按服务端角色检查并记录操作日志。
- 人物证明和认证证书只允许超级/运营管理员按需获取短时查看地址，访问行为进入日志，财务和观察员无权查看。
- 外部观察员只读取汇总快照和自己的分成记录，不存在写接口或任何导出接口。
- 外部观察员响应不包含用户、单笔订单、支付流水、下载明细、COS、证明材料、证书文件或内部日志。
- 自动化检查包含真实单元测试，至少覆盖管理员/观察员边界、运营/财务权限差异和导航入口裁剪；`npm test` 不得只是类型检查别名。
- 真实 COS、支付、短信、邮件、微信和政府认证服务未接入时，只能使用 provider 占位或测试适配器，不能伪装为已接通。

## 12. T009 身份基础验收

- T009 已按本节完成最终检查并通过 PR #19 合并：`npm run lint`、`npm run typecheck`、`npm test`（10/10）、`npm run build` 和 GitHub Actions 均通过。
- 全新环境可按 README 启动 PostgreSQL 16、应用 migration、重复载入非真实 seed 并运行身份 API 测试。
- 注册在同一事务中创建用户、已验证登录身份、`buyer` 基础角色和当前有效条款接受记录。
- 会话 token、CSRF token 和验证码在数据库中只保存哈希；会话 Cookie 包含 `HttpOnly`、`Secure`、`SameSite=Lax`。
- 缺少或错误 CSRF/Origin 的 Cookie 鉴权修改请求必须拒绝。
- 邀请码只能原子激活一次，并同时创建上传者资料和有效 `uploader` 角色。
- `admin` 基础角色和有效后台子角色必须同时存在；超级管理员不自动获得 `buyer` 或 `uploader`。
- `observer` 不能进入管理后台，且只返回自己的合作方摘要，不返回管理员权限。
- `npm test` 只允许重置数据库名以 `_test` 结尾的隔离测试库。
