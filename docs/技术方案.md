## 1. 设计原则

第一版技术方案优先满足：

- 对新手项目管理友好。
- 能快速做出 Web 网站和管理后台。
- 后续容易维护和迭代。
- 尽量减少不必要的平台和工具。
- 先跑通核心业务，再扩展 App、小程序、会员和企业功能。

## 3. 数据库实现建议

数据库实现采用 **Prisma + PostgreSQL**：

- Robert 已确认本地开发和测试使用 **Docker Compose + PostgreSQL 16 + Prisma**。
- T009 建立项目内 Compose 配置、Prisma schema、首个 migration 和最小测试数据；全新环境必须能够按文档重复初始化。
- 本地数据库只使用开发/测试凭证和非真实种子数据，不保存生产密钥或真实用户隐私数据。
- 生产环境继续使用 TencentDB for PostgreSQL，在 T017 按最小权限、备份、恢复和迁移流程接入。
- `docs/数据库设计.md` v2.4 是后续数据模型依据。
- Prisma schema 作为主要表结构表达方式，便于阅读、维护和生成迁移。
- Prisma Migrate 用于记录迁移历史，方便后续 PR 审查。
- 复杂索引、统计视图、物化视图或 Prisma 不方便表达的约束，可用原生 SQL migration 补充。
- T007 阶段只做设计，不安装 Prisma 依赖，不连接真实数据库，不创建生产数据表。

暂不把 Drizzle 作为第一版主方案。Drizzle 更接近 SQL，但当前项目更需要直观、稳定、易审查的模型表达。

### 3.1 T009 本地身份实现

- 项目内 `compose.yaml` 固定 PostgreSQL `16.10-alpine`，默认只绑定 `127.0.0.1:54329`；开发库和测试库分离。
- Prisma 7 使用 `prisma.config.ts`、显式 PostgreSQL driver adapter、`schema.prisma` 和 `prisma/migrations/`；首次 migration 覆盖用户、登录身份、条款接受、基础角色、后台子角色、上传者邀请码、观察员资料、会话和验证码挑战。
- 会话 Cookie 只保存随机 token；数据库只保存 HMAC 哈希。Cookie 使用 `HttpOnly`、`Secure`、`SameSite=Lax`，修改请求同时检查 Origin 和 CSRF token。
- 手机号/邮箱验证码每次随机生成，数据库只保存验证码哈希；本地 provider 把一次性交付写入 Git 忽略且权限为 `0600` 的 `.local` outbox，不写普通日志。
- 手机号是新账号的主身份。手机号验证后可直接注册；邮箱注册必须同时验证邮箱和手机号，再把两个登录身份写入同一用户；若手机号已属于现有账号，则在验证两种凭证后把邮箱绑定到该账号，不重复创建用户。
- 已绑定微信身份的账号可直接登录；首次微信登录必须验证手机号。手机号已存在时把微信身份绑定到原账号，手机号不存在时才创建新账号。真实微信 OAuth/二维码仍在 T017 接入。
- 登录与注册页面共用“验证即登录/注册”状态机：验证码先证明身份，服务端再决定登录、绑定或建号，不能由客户端自行判断账号是否存在。新用户必须接受当前条款；已有用户登录或绑定不重复写条款记录。
- 公开头部只提供一个“登录/注册”认证入口；普通入口携带 `/` 回跳目标，“上传素材”携带 `/upload`。客户端只接受站内相对 `next`，无效目标回退首页；`/register` 作为兼容路由转向统一 `/login`。
- 登录态头部由服务端会话生成用户图标/昵称状态，不依赖 `localStorage`。`/upload` 仍由服务端检查角色：购买用户先进入邀请码激活界面，拥有 `uploader` 角色后才显示素材提交表单。
- 本地 seed 对邀请码采用幂等补充规则：只创建缺失测试码，已有上传者关联永久恢复/保持 `used`，不得清空 `usedByUserId`、`usedAt` 或复活已使用、禁用、过期邀请码；新的重复测试使用独立测试码，不删除旧上传者资料。
- 注册页不要求填写昵称。手机号/邮箱新用户由服务端生成随机昵称；微信新用户优先使用经清理的授权昵称。绑定新登录方式不得覆盖已有账号昵称。
- 身份绑定、验证码消费、账号冲突判断和新用户条款写入必须在同一数据库事务中完成；同一验证码成功后不可重放，两种身份已分属不同用户时禁止自动合并。
- QQ 登录首版不接入，也不在登录/注册页面展示占位入口。
- 本地 provider 需要 `AUTH_PROVIDER=local`；以生产模式运行本地构建时还必须显式设置 `AUTH_LOCAL_ENABLED=true`。腾讯云生产环境不得启用该开关。
- `npm test` 只允许重置名称以 `_test` 结尾的隔离测试库，并真实执行 migration、seed 和身份 API 测试。

### 4.1 T008 API 实现边界

- 核心业务 API 使用 `/api/v1` 前缀，统一 JSON 成功/错误结构和 `requestId`。
- Web 登录使用服务端会话和安全 Cookie；修改请求同时校验来源和 CSRF token。
- 素材文件由应用服务签发短期上传地址，浏览器直传私有 COS，完成后由应用服务查询并核验对象；Next.js 不代理文件正文。
- 支付和退款回调必须基于原始请求体验签、校验金额并幂等处理；客户端支付成功页面不能直接修改订单状态。
- 365 天为平台下载资格，每次实际下载再签发建议不超过 10 分钟的 COS URL。
- 多角度原文件按不可变清单生成一个 ZIP 下载包，再对 ZIP 签发短时 COS URL。
- 会话、验证码挑战、幂等、上传意图和回调事件属于运行时支撑数据，在对应开发任务中补充最小必要模型或受控缓存。
- 外部观察员接口只读汇总快照和自己的分成记录，不提供导出或业务明细接口。
- T009 开始必须建立可运行的本地 PostgreSQL/Prisma 数据层、migration 和测试数据；mock 只用于界面占位，不能作为核心业务验收依据。真实腾讯云服务继续通过 provider adapter 延后接入。

### 4.2 已确认的水印预览与购买交付架构

Robert 已确认第一版采用“预生成独立水印衍生图 + CDN 分发”的方案，不在用户浏览请求中由 Next.js 动态处理原图：

```text
私有 COS 原文件
  ↓ 上传完成并通过对象校验
异步图片处理任务（生产使用腾讯云图片处理能力或等价受控任务）
  ↓
独立水印预览图 + 缩略图（与原文件使用不同 object key）
  ↓
CDN 缓存、压缩和按展示尺寸分发水印衍生图
  ↓
公开素材列表 / 详情页

购买成功 + 授权校验
  ↓
私有原文件清单生成 ZIP
  ↓
应用服务签发 ZIP 的短时下载地址
```

实现边界：

- 原文件、人物证明材料、认证证书和 ZIP 始终私有；公开页面和 CDN 不得访问这些对象。
- 水印预览图和缩略图必须是预先或异步生成的独立文件，不允许仅靠前端叠加水印，也不允许通过修改公开 URL 推导原文件路径。
- CDN 负责缓存、压缩和展示尺寸变换；核心水印在衍生图生成阶段固定写入，不能依赖客户端显示层。
- Next.js 只保存/查询文件元数据、校验权限、返回水印衍生图地址和签发购买后 ZIP 下载；不读取、缓存、解码或处理图片文件正文，因此不会因水印处理显著占用应用服务器内存。
- T010 使用本地种子数据或受控存储适配器提供“已生成”的水印衍生图；T011 通过可替换处理适配器跑通原文件确认后的衍生图生成状态；真实 COS、图片处理和 CDN 在 T017 接入。
- 素材上架前至少一张水印预览图必须生成成功；处理失败时保持不可上架并允许受控重试，不能回退为公开原图。

T010 实现状态：本地受控存储适配器只把白名单 bucket 和不透明水印衍生图 object key 映射为公开路径；Prisma 记录私有原文件和独立公开预览文件，公开查询只选择处理状态为 `ready` 的水印预览/缩略图。该实现验证了架构边界，但不代表真实腾讯云 COS、图片处理或 CDN 已接通。

T011 实现状态：新增 `local_test` 存储与处理 provider，用于在本地验证上传意图、暂存 key 到不可变最终 key 的状态变化、对象元数据核验，以及独立 preview/thumbnail 衍生记录。人物证明使用必填私有 `person_proof`，物件/道具和场景可选填私有 `supporting_proof`；两者都不生成公开衍生图。最终提交后前端切换到独立待支付结果页，真实支付成功后才由 T013 推进到待审核。该 provider 不保存图片正文、不返回内部 bucket/region/object key，也不声称已接通 COS；Next.js 只接收浏览器计算的文件元数据并运行本地任务状态机。生产环境必须在 T017 替换为真实私有 COS 直传、服务端对象查询和异步图片处理，不能直接复用本地测试地址。

T012 实现状态：管理后台新增服务端角色守卫和 CSRF 校验，所有审核、基础信息编辑、认证核验、上架/下架及敏感文件查看均由 Next.js API 调用 Prisma 事务持久化，并写入 `asset_review_events` 或 `audit_logs`。数据库通过约束阻止未审核或未认证素材变为 `listed`；服务层进一步要求完整证书记录和可用 `public_preview`。敏感文件查看使用 5 分钟 HMAC 短时 token，响应不返回 bucket、region 或 object key，且每次请求写入审计日志。本地 `local_test` 认证文件上传仍只验证元数据，不保存文件正文；真实 COS 签名查看和政府认证网站接入继续属于 T017，真实退款执行属于 T013。

## 5. 第一版架构思路

```text
用户浏览器
  ├─ 业务请求 → Next.js 应用服务 → TencentDB for PostgreSQL
  ├─ 上传文件 → 私有腾讯云 COS → 异步生成水印衍生图
  ├─ 浏览图片 → CDN → 水印预览图/缩略图
  └─ 购买下载 → 应用服务校验授权 → 私有 ZIP 短时签名地址
```

微信支付、支付宝、短信、认证、腾讯云 COS 等外部服务应通过接口封装，避免业务代码直接依赖某一个服务商。

## 6. 中国大陆部署结论

当前结论：

- 网站面向中国大陆访问，需要按中国大陆网站要求准备备案。
- Vercel 官方区域列表显示其计算区域包含香港、新加坡、东京等，但没有中国大陆区域；因此不建议把 Vercel 作为中国大陆生产部署默认方案。
- Vercel 可作为早期海外预览或临时演示备选，但生产环境建议使用支持中国大陆备案和大陆节点的云服务商。
- 推荐生产部署方向：腾讯云大陆节点 + TencentDB for PostgreSQL + 腾讯云 COS + CDN/HTTPS + 备案。
