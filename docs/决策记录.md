### D024：T008 核心 API 与安全边界

日期：2026-07-13
当前状态：T008 第一版接口合同已完成全项目审核修订；Robert 于 2026-07-15 批准，并通过 PR #15 合并到 `main`。
决策：

- 第一版业务 API 使用 `/api/v1` 前缀，统一 JSON 成功/错误结构、`requestId`、整数分金额、ISO 8601 UTC 时间和游标分页。
- Web 登录使用服务端会话和安全 Cookie；Cookie 鉴权的修改请求同时校验来源和 CSRF token，不把登录令牌存入 `localStorage`。
- 文件采用“应用服务签发短期上传地址、浏览器直传私有 COS、应用服务确认并核验对象”的方式，Next.js 不代理文件正文。
- 支付与退款回调必须验签、校验商户和金额、幂等去重；购买支付成功后在事务中生成订单状态、授权、下载入口和收益记录。
- 365 天是平台下载资格；每次实际下载只签发建议不超过 10 分钟的 COS URL，不返回 object key 或原始下载 token。
- 外部观察员 API 只读汇总快照和当前合作方分成记录，不提供导出、明细遍历或敏感字段。
- 会话、验证码挑战、幂等记录、短期上传意图和回调去重属于运行时支撑数据，由 T009、T011、T013 在实现阶段补充最小必要模型或受控缓存。

来源：Robert 2026-07-13 确认按推荐登录、COS 直传和支付回调方案继续 T008。
影响：T009-T015 应以 `docs/API.md` 当前版本为接口合同；真实第三方账号、密钥和 provider 配置未接入前，只实现可验证的本地测试 provider 和适配层，不得伪装为已联通。

### D029：T009A 三套视觉方向共同校准与参考网站

日期：2026-07-16
决策：T009A 保留 A `专业创作市场`、B `AI 影像实验室`、C `数字素材档案馆` 三套方向，不取消 AI 影像实验室，也不把三套方案收敛成同一种风格。三套方案共同以现实人物、物件/道具、场景图片和多角度照片为视觉主体，保持正式、专业、简约、认证可信；AI 是主要用户和生产使用场景，可以在 B 方向通过冷色、沉浸感和创作用途语言加强，但不采用过度霓虹赛博、娱乐社区或模型市场风格。

### D032：手机号主身份、微信主入口和邮箱辅助入口

日期：2026-07-17
决策：登录和注册页面把手机号与微信作为顶部主要方式，手机号默认选中；邮箱作为表单下方“其他方式”的独立入口；首版不接入或展示 QQ。手机号验证后可直接注册。邮箱注册必须同时完成邮箱验证和手机号验证，才创建或绑定账号。已绑定微信的账号可直接登录；首次微信登录/注册必须验证手机号，手机号已有账号时绑定到原账号，手机号不存在时才创建新账号。登录方式不是不同账号类型，手机号、邮箱和微信最终必须归属于同一个 `users` 记录。
原因：该层级更符合中国大陆目标用户的常用登录习惯，同时手机号可以承担找回、安全校验和避免重复账号的主身份作用；邮箱仍为需要它的用户保留，但不占据主要入口。QQ 会增加额外接入、授权和账号冲突处理成本，首版暂不做。
影响：T009A 的 `AuthFlowForm`、登录/注册页面说明和视觉验收规则需要调整；T009 身份 API 在不改变 Cookie、CSRF、验证码哈希、条款审计、邀请码和角色守卫的前提下，增加邮箱双验证以及微信首次绑定手机号逻辑。真实短信、邮件和微信 OAuth/二维码仍在 T017 接入，当前页面和本地 provider 不得伪装为已接通。
来源：Robert 2026-07-17 在同类网站调研后明确确认，并指示“以这个决定为准，做相应的修改”。

### D033：验证即登录/注册、服务端昵称和安全绑定规则

日期：2026-07-17
决策：手机号和邮箱界面使用统一的“登录/注册”动作，验证码通过后由服务端判断已有身份登录或新身份建号。新邮箱验证后必须绑定手机号；首次微信使用也必须绑定手机号。手机号/邮箱新账号由服务端生成随机昵称，微信新账号优先采用经清理的授权昵称；注册表单删除“显示名称”。已有用户绑定邮箱或微信时保持原昵称和原条款审计记录。界面不显示“主要方式”标题，只保留手机号、微信顶部入口和下方邮箱入口。
原因：用户在提交验证码前不应承担“我是在登录还是注册”的判断；服务端统一决策可以减少重复账号、账号枚举和前后端状态分叉。昵称不参与身份认证，提前要求填写只会增加注册阻力。
影响：`POST /api/v1/auth/register` 同时承担当前页面的验证码登录/注册入口，已有身份返回 `200`，新建账号返回 `201`；新用户条款校验、验证码消费、身份绑定和冲突判断在同一事务内完成。旧 `POST /api/v1/auth/login` 暂时保留兼容。真实短信、邮件和微信 OAuth/二维码仍在 T017 接入。
来源：Robert 2026-07-17 要求按调研建议修改、删除“主要方式”和注册显示名称，并确保登录/注册无逻辑漏洞。

### D034：单一认证入口、来源回跳和登录态头部

日期：2026-07-17
决策：公开头部把原“登录”和“注册”合并为一个“登录/注册”按钮，并与“上传素材”共用 `/login` 认证页。普通入口携带首页 `/` 作为成功目标；上传入口携带 `/upload`。登录成功后首页头部用用户图标与昵称替换“登录/注册”；`/register` 只保留为旧链接兼容跳转。认证页顶部标签改为“手机号登录”和“微信登录”，手机号状态的条款提示与“登录/注册”主按钮放在“其他方式”之前，邮箱状态的对应控件仍位于邮箱字段之后。
原因：手机号验证码的服务端流程本就同时处理已有用户登录和新用户建号，拆分公开入口会增加选择负担并产生重复页面。来源回跳让普通浏览与上传意图在共用认证页后继续各自任务，同时不能降低上传者邀请码和角色守卫。
影响：无效、站外或缺失的 `next` 统一回退首页；“上传素材”认证完成后进入 `/upload`。普通购买用户没有 `uploader` 角色时不再跳到通用无权限页，而是在 `/upload` 看到邀请码激活界面；激活成功刷新同一路由并进入提交表单。已有上传者直接进入提交表单。真实短信、邮件和微信 OAuth/二维码仍在 T017 接入，本决策不开放无邀请码上传。
来源：Robert 2026-07-17 对 T009A 页面标注和统一登录/注册、上传回跳、登录态首页的明确要求。

### D035：已使用邀请码不可被本地 seed 重新开放

日期：2026-07-18
决策：上传者邀请码一旦已关联上传者资料，即永久按已使用处理；重复运行本地 `db:seed` 只能创建缺失测试邀请码，不得清空使用者、使用时间或把已使用、禁用、过期邀请码恢复为未使用。若历史测试数据出现“邀请码未使用但已有上传者资料”的不一致，seed 根据既有资料恢复已使用状态；激活接口把任何既有使用者或上传者关联返回为 `INVITE_CODE_USED`。新的本地测试流程使用独立邀请码，不删除旧上传者数据。
原因：旧种子脚本每次 upsert 都把固定邀请码恢复为未使用，却保留 `uploader_profiles.invite_code_id` 唯一关联，导致另一个账号激活时触发数据库唯一约束并显示模糊 500。
影响：`prisma/seed.ts` 改为幂等补充与一致性修复，新增本地邀请码 seed helper；激活服务增加遗留关联防御；README、API、数据库、技术和测试文档同步说明。正式后台仍按完整邀请码只展示一次、数据库只存哈希和单次激活规则实现。
来源：Robert 2026-07-18 本地激活测试反馈并授权修复及同步相关内容。
